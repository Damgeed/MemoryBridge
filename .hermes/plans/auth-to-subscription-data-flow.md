# Auth-to-Subscription Data Flow: Proposal

**Architect:** Henry  
**Date:** 2026-05-26  
**Status:** Proposal for Review

---

## 1. Executive Summary

The current auth-to-subscription pipeline has four critical problems:
1. **`/dashboard/welcome`** creates Org+Sub+API key without a User record — orphan data
2. **`/dashboard/free-signup`** creates a User with a random password never shared — unclaimable account
3. **Auth0 users** may lack API key visibility if their org was generated from a different path
4. **Dual credential storage** (JWT + API key in localStorage) with confusing fallback logic

The core issue: **Subscription and User records are not coupled**. A subscription can exist without a User, and a User can exist without a subscription being visible to them.

This proposal redesigns the state machine to enforce: **`anonymous → registered → subscribed`** as a strict linear progression where each state transition is gated and produces atomic, well-coupled records.

## 2. Current State Machine: Problems in Detail

### 2.1 Entry Points Creating Orphaned Data

```
                  ┌──────────────────┐
                  │   ANONYMOUS      │
                  └────────┬─────────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ /welcome │ │free-     │ │ Auth0    │
       │          │ │signup    │ │ password │
       └────┬─────┘ └────┬─────┘ └────┬─────┘
            │            │            │
            ▼            ▼            ▼
      Sub+Key+Org   User+Sub+Key   User+Sub+Key
      NO USER       +Org           +Org
                    random pwd     proper auth
                    (unclaimable)   
```

**Problems per path:**

| Path | Creates User? | User Can Log In? | Has API Key? | Subscription Linked? |
|------|:---:|:---:|:---:|:---:|
| `/dashboard/welcome` | ❌ No | ❌ N/A | ✅ Once | ✅ But orphan |
| `/dashboard/free-signup` | ✅ Yes | ❌ No (random pwd) | ✅ Once | ✅ |
| Auth0 passwordless | ✅ Yes | ✅ Yes (OTP) | ✅ Auto-created | ✅ Free default |
| Auth0 social login | ✅ Yes | ✅ Yes (social) | ✅ Auto-created | ✅ Free default |
| Email/password register | ✅ Yes | ✅ Yes | ❌ Not auto | ❌ Not auto |

### 2.2 Data Model Coupling (Current)

```
User {                    Subscription {               API Key {
  id (UUID)                 id (stripe_id |              id (UUID)
  email (UNIQUE)             "free-{org[:8]}")           key_hash (bcrypt)
  password_hash             organization_id (UNIQUE)     label
  organization_id           stripe_customer_id           project_id → org_id
  auth0_sub                 tier                         is_active
}                           status                     }
                            current_period_start
                            current_period_end
                          }
```

**Problems:**
- `User.organization_id` is a string, not a foreign key — it's the same UUID generated per-path
- `Subscription.organization_id` is also the same UUID, but no FK constraint
- `API Key.project_id` is meant to be the org_id but lacks formal constraint
- **No table has a FK relationship** — all references are string-based and can drift
- Three independent UUID generation sites: `free-signup`, `welcome`, `Auth0 callback`

### 2.3 Checkout Flow (Current)

```
Pricing Page → POST /billing/checkout
    ↓
Check auth state — get org_id from request.state.auth
    ↓
Create Stripe CheckoutSession with client_reference_id=org_id
    ↓
User pays on Stripe
    ↓
Redirect to /dashboard?session_id={id}
    ↓
Frontend polls /dashboard/data until tier != 'free'
    ↓
Stripe webhook: checkout.session.completed
    → store Subscription with org_id from session.client_reference_id
```

**Problem:** The org_id used during checkout was created during the Free Signup/Auth0 registration. If the user was never registered (welcome path), the org_id was generated by the frontend and the Stripe webhook will store a subscription against an org_id that has no User record. The user can never manage their subscription.

## 3. Proposed Ideal State Machine

### 3.1 Core Principle: Registration First, Subscription Second

```
                    ┌──────────────────┐
                    │   ANONYMOUS      │
                    │  (no auth)       │
                    └────────┬─────────┘
                             │
                    (register via Auth0
                     passwordless or social)
                             ▼
                    ┌──────────────────┐
                    │   REGISTERED     │ ← User + Org + Free Sub
                    │  (has JWT)       │      + API Key created
                    └────────┬─────────┘      atomically
                             │
                    (subscribe via Stripe)
                             ▼
                    ┌──────────────────┐
                    │   SUBSCRIBED     │ ← Subscription upgraded
                    │  (has JWT +      │      from free → paid
                    │   API key)       │
                    └──────────────────┘
```

**Invariant:** Every Subscription MUST have exactly one User via its organization_id. Every User MUST have exactly one Subscription (default free).

### 3.2 Data Model (Proposed)

```sql
-- Users table: ONE source of truth for identity
CREATE TABLE users (
    id              TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
    email           TEXT NOT NULL UNIQUE,
    password_hash   TEXT NOT NULL DEFAULT '',   -- empty for Auth0 users
    name            TEXT NOT NULL DEFAULT '',
    organization_id TEXT NOT NULL UNIQUE,       -- 1:1 User↔Org
    auth0_sub       TEXT DEFAULT '',            -- Auth0 subject ID
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- NOTE: organization_id is UNIQUE here — each user owns exactly one org

-- Subscriptions: ONE per organization_id (enforced by UNIQUE on org)
CREATE TABLE subscriptions (
    id                  TEXT PRIMARY KEY,        -- Stripe sub ID or "free-{org[:8]}"
    organization_id     TEXT NOT NULL UNIQUE,    -- FK to users.organization_id
    stripe_customer_id  TEXT DEFAULT '',
    tier                TEXT NOT NULL DEFAULT 'free',
    status              TEXT NOT NULL DEFAULT 'active',
    current_period_start TIMESTAMPTZ,
    current_period_end  TIMESTAMPTZ,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    FOREIGN KEY (organization_id) REFERENCES users(organization_id)
);

-- API Keys: project_id references users.organization_id
CREATE TABLE api_keys (
    id              TEXT PRIMARY KEY,
    key_hash        TEXT NOT NULL UNIQUE,
    label           TEXT NOT NULL,
    project_id      TEXT NOT NULL,               -- FK to users.organization_id
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at    TIMESTAMPTZ,
    FOREIGN KEY (project_id) REFERENCES users(organization_id)
);
```

**Key changes:**
- `users.organization_id` is now `UNIQUE` — one org per user
- `subscriptions.organization_id` has a `FOREIGN KEY` to `users.organization_id`
- `api_keys.project_id` has a `FOREIGN KEY` to `users.organization_id`
- Subscription ↔ User coupling is now enforced at the database level

### 3.3 Registration Endpoint (Single Path)

**Kill:** `/dashboard/free-signup`  
**Deprecate:** `/dashboard/welcome`  
**Consolidate:** ALL registration goes through Auth0 passwordless (`/auth/auth0/passwordless/verify`)

```
POST /auth/auth0/passwordless/verify  { email, code }
    ↓
Auth0 validates OTP, returns id_token
    ↓
BEGIN TRANSACTION:
  1. Find user by email OR create:
     - Generate organization_id (UUID)
     - INSERT INTO users (email, organization_id, auth0_sub, name)
     - INSERT INTO subscriptions (id=f"free-{org[:8]}", organization_id=org, tier="free", status="active")
     - INSERT INTO api_keys (label="default", project_id=org)
     - INSERT INTO oauth_accounts (user_id, provider="auth0-passwordless", provider_user_id=auth0_sub)
  2. Generate JWT with { sub: user.id, email, organization_id }
COMMIT
    ↓
Return { token, api_key, user }
```

**Atomicity guarantee:** A user registration always produces all four records or none. No orphaned data.

### 3.4 Checkout Flow (Proposed)

**Gate:** User must have a valid JWT (i.e., be Registered) before subscribing.

```
Pricing Page → User clicks "Subscribe"
    ↓
Checks localStorage for mb_jwt
  → If missing: show auth modal first (must register)
  → If present: proceed
    ↓
POST /billing/checkout?tier=pro
  Headers: Authorization: Bearer <JWT>
    ↓
Server:
  1. Decode JWT → extract user_id, organization_id
  2. Validate user exists in database
  3. Create Stripe CheckoutSession:
     - client_reference_id = organization_id
     - metadata = { organization_id, user_id, tier }
  4. Return { checkout_url, organization_id }
    ↓
User completes payment on Stripe
    ↓
Redirect to /dashboard?session_id={id}
    ↓
Frontend loads /dashboard/data
  → If tier='free': poll every 2s (max 60s)
  → If tier!='free': show success
    ↓
Stripe webhook: checkout.session.completed
  1. Extract organization_id from metadata (NOT client_reference_id)
  2. Look up user by organization_id
  3. Retrieve full subscription from Stripe
  4. UPDATE subscriptions SET id=stripe_sub_id, tier=..., status=..., stripe_customer_id=...
  5. Ensure user_id in metadata matches user record
    ↓
Done
```

**Key improvement:** The checkout is gated on authentication. If the Pricing page detects no JWT, it shows the auth modal first — the user MUST register before they can subscribe.

### 3.5 Gating Rules

| Action | Anonymous | Registered | Subscribed |
|--------|:---------:|:----------:|:----------:|
| Use playground (demo key) | ✅ | ✅ | ✅ |
| Register (via Auth0) | ✅ | N/A (logged in) | N/A |
| View dashboard | ❌ → show auth modal | ✅ | ✅ |
| Create API key | ❌ | ✅ (free: max 5, paid: more) | ✅ |
| See subscription details | ❌ | ✅ (free tier) | ✅ |
| Upgrade/pay | ❌ → show auth modal | ✅ | ✅ (or cancel) |
| List/revoke keys | ❌ | ✅ | ✅ |
| Use Memory Bridge API | ✅ (demo key) | ✅ (own API key) | ✅ (own API key) |
| Cancel subscription | ❌ | ❌ | ✅ |
| Access admin/export | ❌ | ✅ (based on tier) | ✅ (based on tier) |

### 3.6 Session State Model

```
Session state is derived from one thing: the JWT.

JWT payload:
{
  "sub": "<user_id>",
  "email": "<email>",
  "organization_id": "<org_id>",
  "tier": "free|starter|pro|enterprise",
  "iat": <issued_at>,
  "exp": <expires_at>
}

Frontend stores ONLY in localStorage:
  mb_jwt        → JWT (primary auth token)
  mb_api_key    → Cached API key (derived from JWT on login)

The frontend uses JWT for all dashboard/non-API calls.
The frontend uses API key for Memory Bridge API calls (programmatic access).
```

**Simplification:** Remove the dual-fallback pattern in `getAuthHeaders()`. Use JWT for dashboard API calls. Only use the API key for Memory Bridge data operations (storing/querying memories).

## 4. Migration Path

### Phase 1: Schema Changes (Backward-Compatible)

Add FK constraints as new columns to avoid breaking existing records.

```sql
-- Migration v15: Add FK constraints between existing tables

-- 1. Make subscriptions.organization_id a FK to users
--    (But first, ensure all existing subscriptions have a matching user)
--    For orphaned subscriptions (no user), create placeholder users.

-- Step A: Find orphaned subscriptions
SELECT s.organization_id
FROM subscriptions s
LEFT JOIN users u ON s.organization_id = u.organization_id
WHERE u.id IS NULL;

-- Step B: For each orphaned subscription, find the closest matching user
-- by checking if any user has a related Stripe customer email
-- Or create a "migration placeholder" user
INSERT INTO users (email, name, organization_id, created_at, updated_at)
SELECT
  CONCAT('migrated-', s.organization_id, '@system.local'),
  CONCAT('Migrated User (', s.organization_id, ')'),
  s.organization_id,
  NOW(),
  NOW()
FROM subscriptions s
LEFT JOIN users u ON s.organization_id = u.organization_id
WHERE u.id IS NULL;

-- Step C: Now it's safe to enforce the FK
-- (We use a NOT VALID constraint first to avoid table locks in PG)
ALTER TABLE subscriptions
  ADD CONSTRAINT fk_subscriptions_org
  FOREIGN KEY (organization_id) REFERENCES users(organization_id)
  NOT VALID;
-- Later: ALTER TABLE subscriptions VALIDATE CONSTRAINT fk_subscriptions_org;

-- Step D: Add FK for api_keys
ALTER TABLE api_keys
  ADD CONSTRAINT fk_api_keys_org
  FOREIGN KEY (project_id) REFERENCES users(organization_id)
  NOT VALID;
-- Later: ALTER TABLE api_keys VALIDATE CONSTRAINT fk_api_keys_org;
```

### Phase 2: Deprecate /dashboard/free-signup

1. Add a deprecation warning header to the response
2. After 1 week, change it to redirect to Auth0 passwordless flow
3. After 2 weeks, remove the endpoint entirely

### Phase 3: Deprecate /dashboard/welcome

1. Remove the legacy `welcome_setup` handler
2. The Stripe redirect URL should go to `/dashboard` (which handles the session_id polling)
3. The webhook `checkout.session.completed` handles subscription creation — no need for a separate welcome endpoint

### Phase 4: Add Checkout Auth Gate

Modify `create_checkout_auth` (the authenticated checkout at `/billing/checkout`) to be the **only** checkout path. Remove the unauthenticated `create_checkout` at `/billing/checkout/{org_id}`.

```python
@router.post("/checkout")
async def create_checkout(
    request: Request,
    tier: str = "pro",
    billing: BillingService = Depends(_get_billing_service),
):
    # Require JWT authentication with user_id in payload
    auth = getattr(request.state, "auth", None)
    if not auth or not auth.get("user_id"):
        raise HTTPException(
            status_code=401,
            detail="You must be signed in to subscribe. Please register first.",
        )
    
    user_id = auth["user_id"]
    org_id = auth.get("project_id")
    
    # Verify user exists in database
    user = await storage.get_user_by_id(user_id)
    if not user:
        raise HTTPException(status_code=401, detail="User not found. Please register.")
    
    url = await billing.create_checkout_session(
        organization_id=org_id,
        user_id=user_id,  # NEW: pass user_id in metadata
        tier=tier,
    )
    ...
```

### Phase 5: Fix `_find_sub_by_id`

The current implementation always returns None. Implement it properly:

```python
async def _find_sub_by_id(self, sub_id: str) -> Optional[Subscription]:
    """Find a subscription by its Stripe ID."""
    if not self.repo:
        return None
    try:
        # Need a get_subscription_by_id method on the repo
        return await self.repo.get_subscription_by_id(sub_id)
    except Exception:
        return None
```

Add `get_subscription_by_id` to the `MemoryRepository` interface and implement in both backends.

### Phase 6: Frontend Simplification

1. Remove the dual-fallback in `getAuthHeaders()` — use JWT for dashboard APIs
2. On login, fetch API key from `/auth/my-key-value` and cache in `mb_api_key`
3. If JWT expires, redirect to auth modal (not silently fall back to API key)
4. The pricing page should check `mb_jwt` before allowing checkout

## 5. Edge Cases and Error States

| Edge Case | Current Behavior | Proposed Behavior |
|-----------|:----------------:|:-----------------:|
| User registers, JWT expires | Falls back to API key silently | Show auth modal, require re-login |
| User visits `/dashboard` without auth | Shows blank page with auth modal | Same (but improved error message) |
| Stripe webhook receives event for unknown org_id | Creates subscription anyway (warns) | Log error, send admin alert, skip |
| User has Stripe subscription but no local record | `/dashboard/restore-subscription` creates it | Webhook handles it; restore is fallback |
| Anonymous user clicks "Subscribe" | Goes to Stripe, creates orphan sub | Redirected to auth modal first |
| User deletes account | Subscription orphaned | Set subscription.user_ref to deleted marker |
| Email changes in Auth0 | User record not updated | Auth0 callback updates email on login |

## 7. Summary of Changes

- **Couple User ↔ Subscription** via FK on `organization_id`
- **Single registration path** via Auth0 passwordless (kill `free-signup`)
- **Auth gate on checkout** — must have JWT to subscribe
- **JWT as primary session token** — API key is cached convenience for programmatic use
- **Webhook uses metadata** (not just `client_reference_id`) for reliable org resolution
- **Migration backfills** orphan subscriptions with placeholder users
- **Phase 1** is safe to apply to production — NOT VALID FK constraints don't lock tables
