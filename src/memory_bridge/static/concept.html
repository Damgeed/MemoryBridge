Memory Bridge Architecture

764|

765| Cross-session memory persistence for AI agents — store, retrieve, and share memories across conversations and sessions. 766|

767| 768|

Core Components

769|
    770|
  • 771| STORE 772| Memory Ingestion — Agents write structured memories (facts, decisions, preferences) to a central store via a lightweight API. Each memory carries context, tags, and metadata for precise retrieval. 773|
    • 774|
  • 775| RETRIEVE 776| Contextual Recall — When an agent needs past context, it queries the store with semantic search, keyword matching, or entity graph traversal. Results are ranked by relevance and returned as actionable context. 777|
    • 778|
  • 779| CORE 780| Memory Core — The central persistence layer. Dual-backend (SQLite for dev, PostgreSQL for production) with multi-tenancy via schema-per-tenant and RLS. Redis-backed rate limiting for production safety. 781|
    • 782|
  • 783| ORCHESTRATE 784| Agent Coordination — The orchestrator manages which agent stores what, resolves conflicts (duplicate facts, contradictory preferences), and triggers consolidation when the memory graph grows beyond thresholds. 785|
    • 786|
787| 788|

Data Flow

789|

790| When an agent stores a memory, the payload is validated, indexed by embedding (for semantic search), and persisted with full metadata. 791| Retrieval queries run through a multi-stage pipeline: keyword filter → semantic ranker → entity resolver → dedup → response. 792| Cross-session sharing uses the entity graph — agents can discover related memories created by other agents in the same organization. 793|

794| 795|

Security & Multi-Tenancy

796|

797| Every organization gets an isolated schema. Row-Level Security (RLS) enforces tenant boundaries at the database level. 798| Authentication uses stateless JWTs (HS256, 60-minute expiry) for users and bcrypt-hashed API keys for programmatic access. 799| All endpoints are gated by the API key middleware with 34 exempt paths for public pages and webhooks. 800|
