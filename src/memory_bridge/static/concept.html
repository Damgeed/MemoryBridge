Memory Bridge Architecture

Cross-session memory persistence for AI agents — store, retrieve, and share memories across conversations and sessions.

Core Components

  • STORE Memory Ingestion — Agents write structured memories (facts, decisions, preferences) to a central store via a lightweight API. Each memory carries context, tags, and metadata for precise retrieval.
  • RETRIEVE Contextual Recall — When an agent needs past context, it queries the store with semantic search, keyword matching, or entity graph traversal. Results are ranked by relevance and returned as actionable context.
  • CORE Memory Core — The central persistence layer. Dual-backend (SQLite for dev, PostgreSQL for production) with multi-tenancy via schema-per-tenant and RLS. Redis-backed rate limiting for production safety.
  • ORCHESTRATE Agent Coordination — The orchestrator manages which agent stores what, resolves conflicts (duplicate facts, contradictory preferences), and triggers consolidation when the memory graph grows beyond thresholds.

Data Flow

When an agent stores a memory, the payload is validated, indexed by embedding (for semantic search), and persisted with full metadata. Retrieval queries run through a multi-stage pipeline: keyword filter → semantic ranker → entity resolver → dedup → response. Cross-session sharing uses the entity graph — agents can discover related memories created by other agents in the same organization.

Security & Multi-Tenancy

Every organization gets an isolated schema. Row-Level Security (RLS) enforces tenant boundaries at the database level. Authentication uses stateless JWTs (HS256, 60-minute expiry) for users and bcrypt-hashed API keys for programmatic access. All endpoints are gated by the API key middleware with 34 exempt paths for public pages and webhooks.
